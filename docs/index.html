Introduction

Nodejs Starter Server

This Node.js starter server is designed to be a powerful and flexible foundation for your projects, providing essential functions and robust security to kick-start your development process.

Flexibility is a core feature of this starter server. It allows you to seamlessly add extra fields to any model without modifying the original schema. These additional fields are stored in a separate model, ensuring that the primary structure remains untouched and maintainable. This approach not only preserves the integrity of your original models but also makes it easy to retrieve and utilize these extra fields whenever needed.

This Node.js starter server is an ideal tool for quickly and securely launching your next project, offering a solid, secure, and flexible foundation.

Installation

Follow these steps to install and set up the Node.js Starter Server on your local machine

Requirements

Nodejs V18.18.0

npm V9.8.1

Mongdb database

Installation

Start by cloning the repository to your local machine using Git. Open your terminal and run the following command:

git clone https://github.com/YousefAlsbaihi/NodeJs-Starter-Server.git

This command creates a copy of the repository in a new directory called NodeJs-Starter-Server in your current location, then navigate to the directory

cd NodeJs-Starter-Server

Install all the required dependencies using npm (Node Package Manager). Run the following command in the terminal:

npm install

This command reads the package.json file and installs all the libraries and modules specified under dependencies and devDependencies.

Create a .env file in the root directory of the project to store your environment variables. Use the provided .env.sample file as a template. Copy the sample file to create a new .env file:

cp .env.sample .env

Edit the .env file with your preferred text editor to configure the necessary environment variables (such as database connection strings, JWT secret keys, etc.)

Now you’re ready to start the server in production like mode

npm start

and this command to start in development mode

npm run dev

Custom fields / Extra fields

To use Custom / Extra fields there is no extra programing setup needed, all you have to do is send the extra field details along with the request you are making

for example in the users registeration you can send the key and value and it will be stored and can be updated using the user update api and can be retrived just like the rest of the user details

same thing for Files you can just send the extra field data and it will be stored and retrived when the file is requested


Some key points you need to keep in mind :
  • When creating a key, it can be updated when you submit the update request just submit the key and value for the extra field
  • You can create as much fields as you want for anything
  • You can submit multiple extra fields at once, for example in users you can submit fileds (age, birthday, country, city) all at once
  • each field you submit is tied to the user that it's associated with so adding extra field for one user doesn't mean you added it to all users
  • The extra fields will be shown once you retrive the details about the user or the file

  • Number of files can be uploaded at once

When uploading a file it will be encrypted by default and uploaded to the server storage The file will be encrypted using the encryption type you chose in the .env file and to verify data integrity we added hash “sha256” to the files
To upload file, send a post request to the route

/files/upload

with the fileds

  • files the files to upload
  • token user token submitted as bearer token in the header

Encryption

By default all encryptions using "AES-256-CBC" which assigned along with the key inside the .env file.
however this can be changed to some different type of encryption to "AES-256-CTR"
here is different between the two, also more encryptions types will be added in the next updates.


AES-256-CBC vs AES-256-CTR Comparison
Feature AES-256-CBC AES-256-CTR
Mode Block cipher mode Stream cipher mode
Chaining Yes (each block depends on the previous) No (independent blocks)
IV Random and unique for each encryption Unique (incremented or non-repeating)
Padding Required if plaintext is not block-sized Not required
Performance Slower due to chaining Faster, allows parallelism
Error Propagation Affects all subsequent blocks No error propagation
Security Risks Susceptible to padding oracle attacks Susceptible to keystream reuse attacks
Common Use Cases File encryption, data at rest High-performance encryption, data in transit

Backup

The backup system does only backup for the database, it will create an encrypted json file that will be downloaded to your computer
and can be restored in the system.


Some things you need to know about the backup:
  • The backup file is encrypted with the same encryption key that is used to encrypt files
  • Anyone has the permission mod_backup can perform a backup
  • Restoring a backup require permission mod_restore
  • The system must have the same encryption key as the downloaded backup file in order to decrypt it and restore it
  • The restore process will drop the existing database tables and replace it with the backup file

Backup

To performe a backup you need to do a get request to the route

/mod/backup

You would also need to submit the user token in the header and the user must have the permission mod_backup

This will automatically download an encrypted json file with all content and tables of the current database

Note

Backup files are never stored on the server, after download complete it will be deleted from the server

Database restore from backup

To restore a database backup from json file, you need to do a post request to the route

/mod/restore

and need to submit the following fields:

  • restore file type field with the backup json file
  • token user token submitted as bearer token in the header

Note

After restore complete, the restore file will be deleted from the server

Backup encryption

Backup encryption is using the same encryption as files does, you can read about it here

How it works

Moderation has it's own route and permissions separated from the users permissions

the routes are created to be more detailed and give all informations available about specific model in the system

Users

For users moderation you will be able to list all users and edit them, you will also be able to assign specific permissions to any user and change their profile pictures

Moderation will also allows you to delete and create new users

Files

For files moderation you will be able to list all files uploaded to the system, you will also be able to delete and download them

Backup

For the backup moderation you will be able to download encrypted backup and restore it

Users Moderation



Get all users

to get list of all users you need to send a get request to the route

/mod/users

This will return all users, no query reqiured except when you do pagination you need to pass ( page, limit, sort, order )
page and limit are required otherwise you will get the default limits assigned in the .env file

Sort query will accept these values

  • createdAt
  • updatedAt
  • username

Order query will accept these values

  • asc
  • desc

Example call

/mod/users?page=1&limit=15&sort=createdAt&order=desc


Update users

to update a specific user you need to send a put request with the updated details to the route

/mod/update-user/{userId}

where the {userId} is the id of the user you want to update

You can also submit the extra field for that user and it will be created automatically, or if that field exists for that user then it will be updated.




Create users

Submit the new user details as a post request to the route

/mod/create-user

some fields are required:

  • name
  • email
  • password

You can also submit the extra field for that user and it will be created automatically




Banned users

to banned a user, this means they won't be able to login

to do so, you need to send a put request to the route

/mod/ban-user/{userId}

where the {userId} is the id of the user you want to banned




Delete users

to delete a user, you need to send a delete request to the route

/mod/delete-user/{userId}

where the {userId} is the id of the user you want to delete

Backup and Restore Moderation



Create a backup

to create a new backup, send a get request to the route

/mod/backup

This will return an encrypted downloadable json file

to restore backup from json file, you need to send a post request to the route

/mod/restore

When submitting the json file, it should be encrypted with same encryption method and key that is created with.

to submit it you need to send it as a file name "restore"
